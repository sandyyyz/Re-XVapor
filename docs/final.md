# 现场决赛总结报告

本次现场决赛要求操作系统支持并运行以下典型 Linux 应用程序：

* [测试集链接](https://github.com/oscomp/testsuits-for-oskernel/tree/on-site-final-2025?tab=readme-ov-file)

1. **git（本地功能）**
2. **git（网络相关功能）**
3. **vim**
4. **gcc**
5. **rustc**

并要求能够在以下平台上完成部署与运行：

* **qemu-rv**
* **qemu-la**
* **rv-visonfive2**
* **la-2k1000**

最终结果是系统在现场测试中几乎完全失利。尽管在现场投入了大量精力，尝试完成 **AHCI 驱动** 和 **应用适配** 的工作，但在运行成熟应用的严苛环境下，系统暴露出众多问题，表现出脆弱性和不稳定性。以下总结几个主要问题及其原因。

---

## 主要问题与反思

### 1. 内存错误频发，隐患长期存在

在初赛阶段，部分样例会偶发触发内存错误。当时为了快速通过更多测试，选择性忽略并绕过了这些错误，而没有深入定位和修复。
这种取巧方式为后续开发埋下了巨大隐患。随着内核复杂度的提升，这类错误在现场赛中被频繁触发，且难以调试，严重阻碍了系统的稳定性和扩展性。

---

### 2. 动态链接实现不规范

现场赛要求镜像能够正确支持 **动态链接**。然而，在初赛阶段，我（以及部分往届项目）采用了一种**非标准的实现方式**：通过静态编码的方式写死解释器地址，而不是动态获取解释器路径。
这种做法在初赛短期内可以“凑合”运行，但在现场赛中带来了巨大困难：

* 难以找到并匹配合适的动态链接解释器；
* 大量动态链接程序因解释器无法正常加载而直接运行失败。

---

### 3. 软链接支持不完善

我的 VFS 层实现了软链接功能，但底层文件系统 **lwext4** 在软链接支持上存在缺陷：

* `inode` 类型允许软链接；
* 但 `ext4_file` 类型并不支持；
* `lwext4` 在检查时要求两者配对，导致功能上无法真正支持软链接。

结果是：**动态链接库广泛依赖的软链接机制无法正常使用**，最终导致动态链接程序大面积运行失败。

---

### 4. 内核规范性与健壮性不足

现场观察发现，很多队伍在调试现场赛题时进展轻松顺利，原因在于他们的内核已经过长时间积累，较为完善和健壮。
相比之下，我的内核由于：

* 测试覆盖不足；
* 功能设计缺乏规范；
* 健壮性与容错性不足；

导致支持复杂应用环境几乎不可能：运行过程中不断暴露新的错误，问题修复速度完全跟不上测试进度。

---

### 5. 开发起点的选择问题

实践证明：**不应当从 xv6 直接改造开始**。
xv6 距离一个完整可用的现代内核仍然差距巨大，功能缺失严重，改造成本极高。
更理想的方式是：**站在前人肩膀上，选择更成熟的内核基础进行二次开发**，而不是从一个教学用内核开始“硬改”。

---

### 6. AHCI 驱动开发未能完成

在现场尝试实现 AHCI 驱动时遇到以下问题：

* **QEMU LoongArch 平台文档缺失**：赛事官方未提供 `qemu-system-loongarch` 的详细说明；
* **设备寄存器信息对不上**：通过 PCI 空间读取到的设备参数与桥片手册描述存在不一致；
* **关键参数无法确认**：如 PCI BAR Base Address、PCI Config Base Address 等，均缺乏可靠来源；
* **设备发现机制受阻**：尝试研究 DeviceTree、ACPI、EFI 等协议，试图通过 EFI 接口获取参数，但发现 QEMU 启动时并未传入正确的 EFI System Table。

最终导致 AHCI 驱动无法在现场赛中调试成功。

---

## LoongArch AHCI 调试尝试

在调试过程中，尝试通过 EFI System Table 获取设备信息，示例代码如下：

```c
printf("EFI System Table at %p\n", systbl);
printf("Signature: %p\n", systbl->Hdr.Signature);
printf("NumberOfTableEntries: %d\n", systbl->NumberOfTableEntries);

for (int i = 0; i < systbl->NumberOfTableEntries; i++) {
    printf("Table[%d]: %x\n", i, systbl->ConfigurationTable[i].VendorGuid.Data1);
}
```

然而，实际运行时发现 `systbl` 的内容异常，表项并未正确传递，进一步验证了 QEMU LoongArch 平台在 EFI 支持上的缺失或不完善。

---

## 总结与教训

1. **问题要在早期重视并修复**：不能为了短期成绩而忽略潜在的内存错误。
2. **遵循标准实现**：动态链接、VFS 软链接等功能必须规范实现，否则后期问题会被无限放大。
3. **内核必须充分测试**：没有经过系统化测试的内核，在复杂应用环境下会频繁崩溃。
4. **选择合适的起点**：不应从教学内核改造，而应选择更成熟的内核作为基础。
5. **硬件/虚拟化文档的重要性**：缺乏规范的 QEMU 文档，导致 AHCI 驱动开发举步维艰。

这次失败的经历再次说明：**构建一个能够运行成熟应用的通用操作系统内核，必须要有扎实的基础、规范的设计以及充足的测试**，而不是临时抱佛脚的“补丁式开发”。
